MySQL学习-进阶篇(二)索引

Created2025-04-28|Updated2025-06-24|MySQL学习笔记
|Word Count:3.6k|Reading Time:11mins|Post Views:

介绍

索引（index）是帮助MySQL高效获取数据的数据结构（有序）。

  • 优缺点
优势 劣势
提高数据检索的效率，降低数据库的IO成本 索引列也是要占用空间的
通过索引列对数据进行排序，降低数据排序的成本，降低CPU的消耗 索引大大提高了查询效率，同时却也降低更新表的速度，如对表进行insert、update、delete时，效率降低

索引结构

MySQL的索引是在存储引擎层实现的，不同的存储引擎有不同的结构，主要包含以下几种：

索引结构 描述
B+Tree索引 最常见的索引类型，大部分引擎都支持B+树索引
Hash索引 底层数据结构是用哈希表实现的，只有精确匹配索引列的查询才有效，不支持范围查询
R-tree（空间索引） 空间索引是MyISAM引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少
Full-text（全文索引） 是一种通过建立倒排索引，快速匹配文档的方式。类似于Lucene，Solr，ES
索引 InnoDB MyISAM Memory
B+tree索引 支持 支持 支持
Hash索引 不支持 不支持 支持
R-tree索引 不支持 支持 不支持
Full-text 5.6版本之后支持 支持 不支持

索引分类

分类 含义 特点 关键字
主键索引 针对于表中主键创建的索引 默认自动创建 PRIMARY
唯一索引 避免同一个表中某数据列的值重复 可以有多个 UNIQUE
常规索引 快速定位特定数据 可以有多个
全文索引 全文索引查找的是文本中的关键词，而不是比较索引中的值 可以有多个 FULLTEXT

在InnoDB存储引擎中，根据索引的存储形式，可以分为以下两种：

分类 含义 特点
聚集索引(Clustered Index) 将数据存储与索引放到了一块，索引结构的叶子节点保存了行数据 必须有，而且只有一个
二级索引(Secondary Index) 将数据与索引分开存储，索引结构的叶子节点关联的是对应的主键 可以存在多个
  • 如果存在主键，主键索引就是聚集索引
  • 如果不存在主键，将使用第一个唯一索引作为聚集索引
  • 如果表没有主键，或没有合适的唯一索引，则InnoDB会自动生成一个rowid作为隐藏的聚集索引

🔍 select_type 常见值说明：

值 含义说明
SIMPLE 简单查询，没有子查询或 UNION。
PRIMARY 最外层的 SELECT 查询。
UNION UNION 中的第二个或之后的查询。
SUBQUERY 在 SELECT 或 WHERE 中出现的子查询。
DERIVED 出现在 FROM 子句中的子查询。

🔍 type（访问类型）优劣顺序

类型 说明
const 表中最多只有一行匹配，速度最快。
eq_ref 对于每个来自前面的表的行，组合一个唯一索引。
ref 使用非唯一索引或前缀索引。
range 使用索引范围查找（如 BETWEEN、>）。
index 扫描整个索引。
ALL 全表扫描（性能最差）。

🔍 Extra 字段常见值：

值 含义说明
Using index 使用了覆盖索引（索引中包含查询所需全部字段）。
Using where 使用了 WHERE 条件进行过滤。
Using filesort MySQL 需要额外排序操作（可能性能较差）。
Using temporary 使用了临时表（通常在排序或分组时）。
Using index condition 使用了索引下推（Index Condition Pushdown）。
Using join buffer 使用了连接缓存（通常是嵌套循环连接）。
Impossible WHERE WHERE 条件永远为 false，返回空结果。

索引使用

  • 最左前缀法则

查询中使用索引时，必须从索引的最左开始，连续地使用字段，才能命中索引。如果跳跃了某一列，索引将部分失效（后面的字段索引失效）。

✅ 示例：创建联合索引

查询条件 使用索引字段 是否命中索引 说明
WHERE name = 'Tom' name ✅ 是 使用最左前缀
WHERE name = 'Tom' AND age = 18 name, age ✅ 是 使用了连续字段
WHERE name = 'Tom' AND age = 18 AND gender = 'M' name, age, gender ✅ 是 完全使用索引
WHERE age = 18 无 ❌ 否 非最左字段不能单独使用索引
WHERE gender = 'F' 无 ❌ 否 非最左字段不能单独使用索引
WHERE name = 'Tom' AND gender = 'F' name ✅ 部分 中间断了 age，gender 无法用索引
WHERE name = 'Tom' AND age > 18 name, age ✅ 是 范围查询不影响前面字段
WHERE name = 'Tom' AND age > 18 AND gender = 'F' name, age ✅ 部分 范围查询中断，gender 无法用索引

❗ 注意事项

  • ✅ 最左字段必须出现，才能用索引。
  • ❌ 中间字段缺失，会中断索引使用。
  • ❌ 范围查询会终止后续字段的索引使用。
  • ✅ 使用 =、IN、BETWEEN 可继续索引。
  • ❌ 避免在索引字段上做运算，如 LEFT(name, 1)。

🔍 使用 EXPLAIN 查看是否命中索引

  • 模糊查询

如果仅仅是尾部模糊匹配，索引不会失效。如果是头部模糊匹配，索引失效。

  • or连接的条件

用or分割开的条件，如果or前的条件中的列有索引，而后面的列中没有索引，那么涉及的索引都不会被用到。

  • 数据分布影响

如果MySQL评估使用索引比全表更慢，则不使用索引。

  • 覆盖索引

尽量使用覆盖索引（查询使用了索引，并且需要返回的列，在该索引中已经全部能够找到）减少select *。

  • 前缀索引

当字段类型为字符串(varchar,text等)时，有时候需要索引很长的字符串，这会让索引变得很大，查询时，浪费大量的磁盘IO，影响查询效率。此时可以只将字符串的一部分前缀，建立索引，这样可以大大节约索引空间，从而提高索引效率。

  • 单列索引和联合索引

单列索引：即一个索引只包含单个列

联合索引：即一个索引包含了多个列

在业务场景中，如果存在多个查询条件，考虑针对于查询字段建立索引时，建议建立联合索引，而非单列索引

多条件联合查询时，MySQL优化器会评估哪个字段的索引效率更高，会选择该索引完成本次查询

索引设计原则

索引是提升数据库查询性能的关键手段之一。合理的索引设计可以极大提高查询效率，减少 I/O 成本。以下是 MySQL 索引设计的核心原则

以下是 MySQL（及大多数关系型数据库）中进行索引设计时的详细原则与最佳实践，帮助你在保证查询性能的同时，兼顾写入效率和存储开销。建议将此笔记保存为 Markdown 以便查阅。

1. 明确使用场景

  1. 查询驱动
    • 先分析业务最常用的查询：WHERE、JOIN、ORDER BY、GROUP BY；
    • 针对这些场景设计索引，避免盲目全表扫描。
  2. 读写比评估
    • 事务写入多、更新频繁的表要慎用索引；
    • 读多写少的表可以多加索引以优化查询。

2. 合理选择索引列

  1. 高基数列优先
    • 基数＝不同值的数量／总行数；
    • 高基数（如身份证号、用户名）更容易区分行，可提高索引效率。
  2. 避免低基数列单独索引
    • 如性别（只有 male/female）不宜单独建索引；
    • 可与其他高基数列组合为联合索引。

3. 联合索引（复合索引）策略

  1. 最左前缀法则
    • 联合索引 (A, B, C) 只对查询中按顺序使用前缀的字段生效；
    • 如 WHERE A、WHERE A AND B、WHERE A AND B AND C 都可命中，WHERE B 或跳过 B 的 WHERE A AND C 则只能部分命中。
  2. 列顺序设计
    • 过滤性（选择性）最高 的列放最左；
    • 若多列选择性接近，将经常用于排序或分组的列放前面；
    • 避免频繁变化的列放最左，以减少索引维护开销。
  3. 覆盖索引
    • 如果索引包含查询所需的所有字段，就无需回表（“Using index”）；
    • 能显著提高查询性能。

4. 索引类型选用

  1. BTREE（默认）
    • 适用于大多数等值查询、范围查询、排序。
  2. HASH（Memory 引擎）
    • 只支持等值查询，无法用于范围扫描或排序；
    • 内存型场景下可用。
  3. FULLTEXT
    • 文本搜索专用索引，适用于大段文本的关键字匹配；
    • 配合 MATCH … AGAINST。
  4. SPATIAL
    • 地理空间索引（如 MyISAM、InnoDB 的 GIS）；
    • 适用于地理坐标查询。

5. 前缀索引

  1. 节约空间
    • 对长字符串列（VARCHAR(255)、TEXT）只索引前 N 个字符，如 email(20)；
    • 平衡区分度与存储，推荐前缀长度能覆盖 80%~90% 唯一性。
  2. 局限
    • 不能用于覆盖索引和排序；
    • 只能用于前缀匹配查询。

6. 避免无效索引

  1. 函数/表达式列
    • 如 WHERE LEFT(name,3)='Tom' 会导致索引失效；
    • 可考虑创建生成列（Generated Column）并为其建索引。
  2. 类型不匹配
    • 查询常量类型与列类型应一致，避免隐式转换（如 WHERE id='123' 对 INT 列）。
  3. 过多索引
    • 每个索引都会增加写入、更新和删除的开销；
    • 定期 SHOW INDEX、分析慢查询，删除长期未使用的索引。

7. 索引维护与监控

  1. 定期分析
    • 使用 ANALYZE TABLE 更新统计信息；
    • 用 EXPLAIN 分析查询是否命中了索引。
  2. 滚动优化
    • 大表批量改写时先删除索引，批量完成后再重建索引；
    • 减少写入锁等待和 IO 压力。
  3. 慢查询日志
    • 开启慢查询日志，找出未走索引或效率低下的 SQL；
    • 根据日志结果优化索引。

8. 实践总结

  • 以业务为导向：先看用例，再设计索引；
  • 最左前缀：联合索引务必遵循；
  • 覆盖与前缀：能覆盖就覆盖，能节省就前缀；
  • 权衡存储与性能：索引越多，查询越快，但写越慢；
  • 持续迭代：定期审视、调整索引方案。
